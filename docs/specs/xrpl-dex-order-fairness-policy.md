# XRPL DEX Order Fairness Policy

Status: controlled-testnet policy gate
Date: 2026-05-20
Scope: DEX-003 ordering and MEV policy before offer execution implementation

This policy freezes the first DEX ordering rule for controlled testnet. It is a
determinism and claims boundary, not a claim that the public-network DEX is MEV
resistant.

## Policy Decision

Controlled-testnet DEX v1 uses deterministic finalized batch order for
cross-transaction priority, and deterministic book order for fills inside each
transaction.

The policy is intentionally conservative:

- no priority fees;
- no fee-based transaction sorting;
- no off-ledger matcher;
- no local wall-clock ordering in execution;
- no public fairness or MEV-resistance claims.

DEX writes remain gated to controlled write edges until a stronger public-market
ordering policy is designed, implemented, and evidenced.

## Threat Model

The controlled testnet accepts these risks as engineering tradeoffs:

- a write edge can observe submitted orders before they finalize;
- a validator/proposer can influence batch order if it controls batch
  construction;
- network latency can affect which order reaches the write edge first;
- deterministic batch order does not prevent sandwiching or back-running.

The policy rejects these behaviors for v1 implementation:

- protocol-level priority fees that let traders buy earlier execution;
- batch construction that reorders DEX transactions to maximize validator,
  operator, or maker/taker profit;
- hidden fills or fills generated by non-consensus services;
- public claims that controlled-testnet ordering is fair in adversarial public
  markets.

## Consensus Rule

Execution processes DEX transactions in the exact order committed in the
transparent transaction batch. Given the same batch bytes and prior ledger
state, every validator must produce the same offers, fills, receipts, account
history rows, and final state roots.

Inside one `offer_create`, the engine scans the opposing book by:

1. best taker price;
2. lower `created_height`;
3. lower `owner_sequence`;
4. lexicographic `offer_id`.

The engine never consults mempool arrival time, local validator clock, peer id,
network source, or process iteration order while executing a finalized batch.

## Controlled Write-Edge Rule

Before public-market ordering work exists, DEX write RPC must stay behind the
controlled write-edge policy. A controlled write edge should assign an
admission index to accepted DEX transactions and preserve FIFO order within its
own accepted stream, subject to sender sequence validity.

The admission index is not a public fairness proof. It is an operator diagnostic
that makes controlled-testnet behavior auditable. Batch construction should
preserve this order for DEX transactions unless a transaction is invalid,
expired, or blocked by sender sequence.

If multiple controlled write edges are enabled, DEX writes must remain disabled
until there is a deterministic merge policy. The merge policy must be part of a
future DEX ordering upgrade, not an implicit implementation detail.

## Fee Policy

DEX fees pay for state expansion, matching compute, receipt bytes, and replay
cost. They do not buy priority.

Rules:

- `offer_create` and `offer_cancel` fee quotes must not include priority tiers;
- higher fees do not move a transaction earlier in batch order;
- underpaid transactions are rejected, not deprioritized;
- validators do not receive spread, rebate, or notional-based execution reward.

## Receipt And Audit Requirements

Every DEX receipt must expose enough ordering data for audit:

- block height;
- transaction index inside the finalized batch;
- receipt index;
- taker transaction id;
- created offer id, if any;
- maker offer id for each fill;
- fill index;
- maker/taker accounts;
- sent and received amounts by asset;
- fee charged to the transaction source;
- terminal maker offer state when an offer closes.

Account history rows for maker accounts must include the taker transaction id
and finalized batch transaction index so a maker can reconstruct why a fill
occurred without relying on off-ledger logs.

## Mempool Requirements

Mempool admission must be deterministic and replay the same state constraints
as execution:

- sender sequence including pending sender transactions;
- pending offer locks and spendable balances;
- trustline authorization and freeze state;
- max-cross cap from `DEX-002`;
- minimum PFT reserve after fees and locks;
- serialized transaction and receipt size bounds.

Mempool acceptance does not guarantee final execution priority. Only finalized
batch order does.

## Public Claims Boundary

Controlled-testnet DEX v1 may claim:

- deterministic replay for a committed batch order;
- bounded matching cost;
- no priority-fee ordering;
- auditable batch and fill positions.

It must not claim:

- front-running resistance;
- sandwich resistance;
- proposer neutrality;
- equal latency access;
- public-market fairness.

Those claims require a later mechanism such as frequent batch auction,
commit-reveal, encrypted order flow, threshold-decrypted batches, or another
explicitly specified public ordering design.

## Implementation Consequences

The first DEX execution implementation must:

- archive transaction order as consensus data;
- keep offer matching purely deterministic from ledger state and batch order;
- reject or remove unfillable maker offers with stable receipt codes;
- keep fee quotes separate from ordering priority;
- expose DEX ordering fields through `tx` and `account_tx`;
- add replay tests proving same batch order gives same fills and state roots;
- add adversarial tests showing higher fee does not change DEX ordering.

No DEX implementation should merge until those consequences are covered by
focused tests and evidence under `reports/xrpl-feature-parity-dex-*`.

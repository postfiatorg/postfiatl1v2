# Wallet Exchange Custody Model

Status: controlled-testnet model v0
Date: 2026-05-21

This document specifies the minimum exchange/custody model for transparent
ML-DSA accounts in controlled testnet. It is a product and operations boundary,
not a claim that custodian-grade key management, hardware signing, or account
history indexing is complete.

## Account Semantics

PostFiat transparent accounts are address-key accounts:

- an address is derived from an ML-DSA public key;
- a funded recipient account may exist before its public key is bound on-chain;
- the first spend from that account must carry the ML-DSA public key and a
  valid signature from that key;
- once the sender account has a `public_key_hex`, later spends must use the
  same public key;
- a different key for an already-bound account is rejected.

Code references:

- account public key field: `crates/types/src/lib.rs`;
- first-spend key binding and later key mismatch rejection:
  `crates/execution/src/lib.rs`;
- SDK identity and quote-bound signing: `crates/rpc_sdk/src/lib.rs`.

## What Does Not Carry Over From BIP32

ML-DSA does not give PostFiat a BIP32-style public child derivation model. For
controlled testnet, custodians must not assume:

- one public root can derive unlimited deposit addresses;
- watch-only deposit address generation can happen without custody-side secret
  material;
- hardware-wallet or exchange infrastructure can reuse existing secp256k1/xpub
  workflows unchanged.

The safe v0 posture is: every deposit address is generated by the custodian
inside its custody boundary, backed up, mapped to a customer or internal ledger
account, and then watched by address.

## Deposit Address Model

Use unique transparent addresses per customer, subaccount, or deposit intent.
Do not use a shared omnibus address for customer deposits in v0 because the
transparent transfer envelope has no memo/tag field.

A deposit registry entry should include at minimum:

- customer or internal account id;
- chain id;
- address;
- ML-DSA public key hex from `wallet_identity_from_backup`;
- wallet backup custody location or KMS reference;
- account index or internal derivation label;
- created-at timestamp;
- status: `active`, `disabled`, `swept`, or `recovery_hold`;
- last observed balance and last observed ledger height;
- optional first-spend binding status.

The backup or seed material is private. The address and public key are safe to
store in watch-only systems, but the public key alone cannot spend.

Mnemonic-backed wallet creation is a planned UX layer over the same custody
boundary, not a replacement for custody controls. See
`docs/specs/wallet-mnemonic-design.md` for the design. Mnemonic words, BIP39
seed bytes, passphrases, and derived `master_seed_hex` must be treated as
private key material.

## Deposit Detection

Controlled-testnet v0 supports address balance polling and tx/finality
verification. It does not yet provide a production account-history indexer.

Minimum viable attribution flow:

1. Generate a unique address for the customer/deposit intent.
2. Persist the address-to-customer mapping before showing the address.
3. Poll `account` for the assigned address.
4. When balance increases, reconcile the delta against the customer mapping.
5. If the inbound transaction id is known from the sender or an indexer, query
   `tx` and require a confirmed finality summary before crediting.
6. For controlled testnet, require conservative confirmation policy: credit
   only after finality RPC validation succeeds against an allowed endpoint.

Known gap: without account-history indexing, a watcher that only sees a balance
delta cannot independently list every inbound transaction by address. The
controlled-testnet model is acceptable for assigned-address pilots, but
exchange-grade production needs account history or an indexer role.

## Withdrawal Model

Withdrawals are signed from the custody address using the SDK quote-bound flow:

1. Restore public identity from the wallet backup.
2. Request a fee quote with `from` set to that identity address.
3. Decode and validate the quote response.
4. Sign with `wallet_sign_transfer_from_quote`.
5. Submit the signed transfer through an RPC endpoint that permits writes.
6. Decode submit response and poll `tx` finality by submitted tx id.
7. Persist the finality proof id, block hash, certificate id, registry root,
   receipt code, and receipt message.

The SDK rejects quote-sender mismatches and chain-id mismatches with the wallet
backup.

## Watch-Only Constraints

Watch-only systems can safely hold:

- address;
- public key hex;
- customer/internal attribution;
- observed balances;
- tx ids and finality summaries;
- registry-root-backed finality metadata.

Watch-only systems cannot:

- derive new deposit addresses from a public root;
- recover a lost backup from the public key;
- sign a withdrawal;
- infer all historical deposits without an account-history source;
- prove future spend authority for an unbound funded account unless the custody
  system preserved the matching backup.

## Recovery And Key Loss

For controlled testnet, recovery is custody-process recovery, not protocol
recovery:

- losing the backup or seed for an address means losing spend authority;
- the chain cannot rotate an account key after custody loses the original key;
- backups must be encrypted and tested before an address is shown to a user;
- a deposit address with uncertain backup status must be disabled immediately;
- sweeping from old deposit addresses should use ordinary signed transfers into
  a fresh custody address.

Protocol-level account key rotation remains outside this v0 model and is tracked
separately under wallet key-rotation UX.

## Controlled-Testnet Exit Criteria

For the first controlled testnet, the exchange/custody model is credible when:

- unique deposit address assignment is documented;
- SDK create/restore/sign flow exists and is tested;
- public RPC `account`, `transfer_fee_quote`, submit, and `tx` finality paths
  are available;
- finality summaries include registry-root-backed compact certificates;
- operators understand that account-history indexing is not yet production;
- the public claims checklist does not imply hardware-wallet, xpub, or
  exchange-grade custody support.

## Remaining Production Work

- Account-history indexer or archive-backed deposit scanner.
- Hardware-wallet or external signer interface for ML-DSA.
- Formal backup encryption and recovery standard.
- BIP39 mnemonic create/import implementation and custody recovery policy.
- Account key-rotation transaction and wallet UX.
- Custodian audit packet covering dependency versions, KATs, key lifecycle,
  signing policy, finality policy, incident handling, and endpoint policy.
